# Защита от сбоев парсинга: Что происходит если свет выключится?

## TL;DR (Кратко)

**Система ИДЕМПОТЕНТНА**: если процесс упадёт в любой момент, при следующем цикле парсинга всё синхронизируется правильно.

```
Сбой в цикл N → лишние товары в БД
Цикл N+1    → товары синхронизируются ✅
```

## Сценарии сбоев и восстановление

### Сценарий 1: Упал ПЕРЕД upsert_products
```python
fetch_products_details(uuids)  # ← УПАЛ (свет выключился)
    ❌ products = None/[]
if products:  # FALSE
    upsert(...)     # ← Не вызвано
    delete(...)     # ← Не вызвано

# Результат: БД не изменилась ✅ БЕЗОПАСНО
# При следующем цикле: всё обновится
```

### Сценарий 2: Упал ПОСЛЕ upsert ДО delete
```python
upsert_products(products)  # ✅ Товары A,B,D добавлены/обновлены
                           # ← УПАЛ (свет выключился)
delete_products_not_in_uuids(...)  # ❌ Не вызвано

# Результат в БД: A, B, C (лишний), D
# При следующем цикле:
#   1. fetch возвращает A, B, D снова
#   2. upsert обновит их
#   3. delete УДАЛИТ C ✅ Синхронизируется
```

### Сценарий 3: Упал ПОСЛЕ delete
```python
upsert_products([A, B, D])  ✅
delete_products_not_in_uuids([A, B, D])  ✅ Удалён C
                           # ← УПАЛ (хоть и редко)

# Результат: БД уже синхронна ✅ Идеально
```

## Ключевая защита: пустой список UUID

```python
def delete_products_not_in_uuids(self, category_id: str, current_uuids: list[str]) -> int:
    if not current_uuids:
        return 0  # ← ЗАЩИТА: не удаляет при пустом списке
```

**Почему это важно:**
- Если `fetch_products_details()` упадёт → `uuids = []`
- Без защиты: удалит ВСЕ товары категории (катастрофа!)
- С защитой: ничего не удаляет, товары сохранены ✅

**Trade-off:**
```
⚖️ Защита от удаления всех товаров   vs   Лишние товары остаются временно
✅ Выбираем это                           ❌ Это хуже
```

При следующем цикле товары синхронизируются, поэтому временные лишние товары — не проблема.

## Как система восстанавливается?

### Использование `category_state` с хэшем UUID

```python
# Цикл N: спарсили A, B, C
uuid_hash = md5("A,B,C")
db.update_category_state(cat_id, "Category", 3, ["A", "B", "C"])

# Цикл N+1: спарсили A, B, C (без изменений)
uuid_hash = md5("A,B,C")  # ← ОДИНАКОВЫЙ ХЭШИ
# → Код знает что состав не изменился
# → Не нужно обновлять товары заново

# Цикл N+2: спарсили A, B, D (C продан)
uuid_hash = md5("A,B,D")  # ← ДРУГОЙ ХЭШ
# → Код знает что состав изменился
# → Вызывает delete_products_not_in_uuids
```

## Полный сценарий с двумя сбоями

```
1. Исходное состояние:  [A, B, C]

2. ЦИКЛ 1: Сайт вернул [A, B, D] (C продан)
   - upsert([A, B, D])  ✅
   - delete([A, B, D])  ❌ SWЕТ ВЫКЛЮЧИЛСЯ
   - БД: [A, B, C, D] (лишний C)

3. ЦИКЛ 2: Fetch упал
   - fetch()            ❌ СВЕТ ВЕРНУЛСЯ, но fetch упал
   - products = []
   - upsert([])         ✗ Не вызвано
   - delete([])         ✗ Не вызвано (защита)
   - БД: [A, B, C, D] (без изменений)

4. ЦИКЛ 3: Успех
   - fetch()            ✅ Вернул [A, B, D]
   - upsert([A, B, D])  ✅
   - delete([A, B, D])  ✅ УДАЛЁН ЛИШНИЙ C!
   - БД: [A, B, D] ✅ СИНХРОННО с сайтом
```

## Почему система идемпотентна?

**Идемпотентность** = результат одинаковый при повторном выполнении

```python
# Свойства системы:
1. ✅ upsert_products() - INSERT OR REPLACE
   → Можно вызвать 100 раз, результат один (товар либо создан, либо обновлён)

2. ✅ delete_products_not_in_uuids() - SELECT UUID NOT IN (list)
   → Можно вызвать 100 раз, удаляются только товары которых нет в списке

3. ✅ Защита при пустом списке
   → Не случайно удаляем всё при сбое

# Результат:
def sync_category(uuids):
    upsert(fetch_products(uuids))
    delete_products_not_in_uuids(category_id, uuids)

# Можно вызвать эту функцию 1 раз или 100 раз
# Результат будет одинаковый ✅ ИДЕМПОТЕНТНО
```

## Что НЕЛЬЗЯ потерять при сбое?

```python
# ЗАЩИЩЕНО (восстанавливается):
✅ Товары в БД (они обновляются при следующем цикле)
✅ Изменения цен (логируются отдельно)
✅ Telegram уведомления (отправляются после синхронизации)

# НЕ ЗАЩИЩЕНО (теряется):
❌ Telegram уведомления которые были в памяти (all_price_changes)
    → Но это не критично, есть логирование

# Можно улучшить:
💡 Сохранять price_changes в БД между циклами (если важно не потерять уведомления)
```

## Итог

✅ **Система безопасна при сбоях**
- При упадке процесса товары не удаляются случайно
- При следующем цикле всё синхронизируется
- Идемпотентность гарантирует что можно вызвать парсер многократно

⚠️ **Но есть trade-off:**
- Временно могут остаться лишние товары в БД (на 1 цикл)
- Это нормально и безопаснее чем потеря данных
